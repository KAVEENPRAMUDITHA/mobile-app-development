Long-term societal implications of widespread AI hiring systems

Using AI systems in hiring can change how people get jobs in the future. There are some good effects and also some risks.

On the positive side, AI can help companies save time by quickly checking thousands of job applications. It can also reduce some human bias, making the process more fair if it is designed well. AI can help employers find the best candidates based on skills and qualifications, instead of personal opinions.

However, there are also long-term concerns. AI learns from past data, and if that data contains unfair patternslike gender or race biasit can repeat those same mistakes. This could make it harder for some groups of people to get jobs. Another problem is that AI cannot easily understand human qualities like creativity, communication, or emotion. As a result, some good candidates might be rejected unfairly.

There is also the issue of transparency. Many AI systems are not clear about how they make decisions, so applicants may not know why they were rejected. This could reduce trust between people and companies. Also, if hiring becomes too automated, human resource professionals may lose their jobs, and the process may feel less personal.

In the long run, society needs to balance technology and fairness. To make AI hiring helpful, it should be regularly checked for bias, be transparent, and include human judgment when needed.

def remove_stopwords(text_list):
    """Remove common English stopwords."""
    from spacy.lang.en.stop_words import STOP_WORDS
    cleaned = []
    for t in text_list:
        tokens = [word for word in t.split() if word not in STOP_WORDS]
        cleaned.append(" ".join(tokens))
    return cleaned

def normalize_text(text_list):
    """Handle punctuation, case, and whitespace consistently."""
    cleaned = [re.sub(r'\s+', ' ', re.sub(r'[^\w\s]', '', t)).strip().lower() for t in text_list]
    return cleaned

    entities = []

    for t in text_list:

    return numbers

        #  entity  ** !

    return entities_list

def extract_numbers(text_list):
    """Extract potential order numbers (sequences of 5+ digits)."""

        if not nums:
             nums = re.findall(r'\d{5,}', t)

    return numbers_list

  1. Text Cleaning (normalize_text function) Action:

Converted all text to lowercase.

Removed punctuation marks (keeping only letters, numbers, and spaces).

Normalized whitespace (replaced multiple spaces with a single space and removed leading/trailing spaces).

Justification: This step ensures consistency across the dataset. Lowercasing treats words like "Great" and "great" as the same, which is crucial for tasks like word counting and machine learning. Removing punctuation and normalizing spaces reduces "noise" in the text, making it cleaner for algorithms to process accurately. We chose normalize_text over simpler functions because it handles all three aspects (case, punctuation, spacing) comprehensively. Stopwords and numbers were not removed at this initial stage, as they might contain useful information for subsequent steps like entity extraction (order numbers) or sentiment analysis (e.g., the word "not").

  1. Named Entity Extraction (extract_entities_spacy, extract_numbers) Action:

Used spaCy (extract_entities_spacy) to identify specific named entities like Organizations (ORG), Products (PRODUCT), Locations (GPE/LOC), Persons (PERSON), and Dates (DATE).

Used a targeted Regular Expression (extract_numbers) to find potential Order Numbers (sequences of 5+ digits, often following "#" or "Order").

Justification: This step was performed on the original, uncleaned text. Cleaning (like lowercasing or removing '#') would destroy the patterns spaCy and the regex rely on for accurate identification. Extracting these entities helps us understand the context of the feedback what product is being discussed, which seller is mentioned, what order number is referenced. This information is vital for linking feedback to specific transactions or identifying trends related to certain products or sellers.

  1. Language Detection (detect_language_langdetect) Action:

Used the langdetect library to identify the language of each individual feedback entry. Error handling was added for very short texts or detection failures.

Justification: The dataset contains feedback in multiple languages (English, Sinhala, Tamil, French, etc.). Detecting the language is a crucial first step before applying language-specific NLP tools. For example, an English sentiment analysis model should only be applied to English text. Language detection also allows for potentially routing feedback to support staff fluent in that language. This was also performed on the original text as cleaning might sometimes interfere with accurate language detection, especially for shorter comments.

  1. Bag-of-words (BoW) Models
  • Performance : The BoW model's performance is quite limited. It only counts the frequency of words and completely ignores the context and grammer of a sentence. For example , it would fail to understand the differnce between 'good' and 'not good'. As we saw from the classification report, its accuracy is low, often around 50%, which is no better than random guessing.
  • Applicability: This model is simple and fast to train. It can be useful as a basic starting point for very simple text clasification tasks. However it is not suitable for any real-world application where understanding the true sentiment of a customer is important.
  1. Pre-trained transformer (Hugging Face)
  • Performance: The transformer model performs siginificantly better. Because it has been trained on massive amount os diverse text from the internet, it understands grammar, context and the subtle nuances of human language. It correctly identified the sentiment in multiple languages and complex sentences, leading to much higher accuracy and reliability.
  • Applicability: This is the preferred approach for almost all modern sentiment analysis tasks. It is incredibly powerful and provides trustworthy results, even without a large, custome labeled dataset. Its ability to understand context makes it the far more applicable and valuable choice for a business that wants to gain real insights from customer feedback.

def handle_missing_values(df, strategy="mean", column=None):
    """
    Handle missing values with different strategies:
    - 'mean' / 'median' / 'mode' for imputation
    - 'drop' to remove missing rows
    """
    if column is None:
        print("Please specify a column.")
        return df
    
    if strategy == "mean":
        df[column].fillna(df[column].mean(), inplace=True)
    elif strategy == "median":
        df[column].fillna(df[column].median(), inplace=True)
    elif strategy == "mode":
        df[column].fillna(df[column].mode()[0], inplace=True)
    elif strategy == "drop":
        df.dropna(subset=[column], inplace=True)
    else:
        print("Unknown strategy. Options: 'mean', 'median', 'mode', 'drop'")
    
    return df


def detect_outliers_iqr(df, column):
    """Detect outliers in a numerical column using IQR method."""
    Q1 = df[column].quantile(0.25)
    Q3 = df[column].quantile(0.75)
    IQR = Q3 - Q1
    outliers = df[(df[column] < (Q1 - 1.5*IQR)) | (df[column] > (Q3 + 1.5*IQR))]
    print(f"Found {len(outliers)} outliers in {column}")
    return outliers

def treat_outliers(df, column, method="cap"):
    """
    Treat outliers in a numerical column.
    - 'cap' : Cap values at 5th and 95th percentiles
    - 'remove' : Remove rows with outliers
    """
    if method == "cap":
        lower = df[column].quantile(0.05)
        upper = df[column].quantile(0.95)
        df[column] = np.where(df[column] < lower, lower, df[column])
        df[column] = np.where(df[column] > upper, upper, df[column])
    elif method == "remove":
        outliers = detect_outliers_iqr(df, column)
        df = df.drop(outliers.index)
    else:
        print("Unknown method. Options: 'cap', 'remove'")
    return df

#Detect and treat outliers
def detect_outliers_iqr(df, column):
    if column in df.columns and pd.api.types.is_any_real_numeric_dtype(df[column]):
        #Detect outliers in a numerical column using IQR method."""
        Q1 = df[column].quantile(0.25)
        Q3 = df[column].quantile(0.75)
        IQR = Q3 - Q1
        outliers = df[(df[column] < (Q1 - 1.5*IQR)) | (df[column] > (Q3 + 1.5*IQR))]
        print(f"Found {len(outliers)} outliers in {column}")
        return outliers
    else:
        print(f"Column '{column}' is not a valid numerical column")
        return pd.DataFrame()

def treat_outliers(df, column, method="cap"):
    """
    Treat outliers in a numerical column.
    - 'cap' : Cap values at 5th and 95th percentiles
    - 'remove' : Remove rows with outliers
    """
    if method == "cap":
        lower = df[column].quantile(0.05)
        upper = df[column].quantile(0.95)
        df[column] = np.where(df[column] < lower, lower, df[column])
        df[column] = np.where(df[column] > upper, upper, df[column])
    elif method == "remove":
        outliers = detect_outliers_iqr(df, column)
        df = df.drop(outliers.index)
    else:
        print("Unknown method. Options: 'cap', 'remove'")
    return df

    """
    Train a model based on choice:
    - 'logistic' : Logistic Regression
    - 'random_forest' : Random Forest
    - 'svm' : Support Vector Machine
    """
    if model == "logistic":
        clf = LogisticRegression(max_iter=500)
    elif model == "random_forest":
        clf = RandomForestClassifier(n_estimators=100, random_state=42)
    elif model == "svm":
        clf = SVC(probability=True, random_state=42)
    else:
        print("Unknown model. Options: 'logistic', 'random_forest', 'svm'")
        return None
    
    X_train, X_test, y_train, y_test = train_test_split(
        X, y, test_size=0.3, random_state=42, stratify=y
    )
    clf.fit(X_train, y_train)
    y_pred = clf.predict(X_test)
    y_proba = clf.predict_proba(X_test)[:,1] if hasattr(clf, "predict_proba") else None

ML can greatly help in health care, but it also has several risks and challenges that need to be handled carefully.

  • When an AI system makes a mistake, it is hard to know who should be blamed.

  • The sigmoid function changes values so they stay between 0 and 1. This works well for binary classification output layers, it is poor choce for hidden layers in deep networks. It suffers from "vanishing gradient" problem, where gradients can become extremely small, effectively stopping the model from learning. The modern standard for hidden layers is relu (Rectified Linear Unit), which is more computationally efficient and less issues.
  1. Bug2: Excesssively high Dropout Rate (Dropout(0.8))
  • Drop out is a regularization technique which prevents the kodel from overfitting,. It randomly turns off some neurons while training, so the kodel doesn't rely too much on training data.A rate of 0.8 meand 80% of the neurons are deactivated at each step. This rate is extremely high, which is preventing it from learning any meaningful patterns. It is like trying to learn a complex subject while forgettinh 80% of what you just saw. A typical dropout rate between 0.2 and 0.5.
  1. Bug3: Incorrect Activation function for the Output Layer (activation='relu')
  • This is a multi-class classification problem (predicting one of 3 disease stages).The final layer must output a probability distribution across all possible classes, where the probabilities sum to 1. The relu function outputs values from 0 to + infinity and is not suitable for this. The correct activation function for multi-class classification is softmax.
  1. Bug4: Inappropriate Loss Function and optimizer (loss='mean_squared_error', optimizer='sgd')

  • Loss: mean_squared_error(MSE) is a loss function for regression tasks (predicting continuous values). For classification task with integer labels (like 0,1,2), the correct loss function is sparse_categorical_crossentripy.

  • Optimizer: sgd(Stochastic Gradient Descent) is a very basic optimizer. Modern optimizers like adam are generally far more effective as they adapt the learcning rate during training, leading to faster convergence and better performance.

Sometimes, the AI system will receive patient data that looks very different from the data it was trained on. This situation is called an Out-of-Distribution (OOD) case. In such cases, the models predictions are not reliable so it should not try to guess.

The system must be built to detect these unusual cases and handle them safely.

Step 1: Detecting Unfamiliar Data

The Reasoning & Uncertainty Engine checks how confident the model is in its prediction.

When the model sees new or strange data, it cant find a familiar pattern.

It gives very low confidence scores for all possible diseases (for example, Disease A: 10%, Disease B: 8%, Disease C: 12%).

The system notices that the top confidence (12%) is much lower than the safety threshold (like 90%).

It then flags the case as unfamiliar or Out-of-Distribution.

Step 2: Acting Safely (Fail-Safe Response)

Once the case is flagged:

The AI stops itself from giving a diagnosis it does not make a random guess.

Instead, it sends an alert message to the doctor saying: This patients data is unusual and doesnt match the systems training patterns. A reliable AI diagnosis cannot be made. Please review manually.

This ensures patient safety and avoids false or risky predictions.

Step 3: Learning from New Cases

The unusual case is saved in a special database.

A doctor reviews it and enters the correct diagnosis (for example, This was a rare condition X).

That verified case is later used to retrain the model, so the AI can learn from it and recognize similar cases in the future.

In Simple Terms

When the AI faces data it hasnt seen before, it should:

Recognize that its unsure.

Stop and alert a doctor instead of guessing.

Learn from the doctors feedback to improve over time.
